Feature Driven Development

O Feature Driven Development é uma metodologia ágil que organiza o desenvolvimento de software por progresso em recursos. Recursos esses que são similares ao user stories do Scrum (pequena unidade de desenvolvimento designada para completar uma meta dentro do produto). É extremamente prático para projetos complexos e de longa construção e times que procuram uma metodologia simples mas bem estruturada. Ele foi projetado para seguir um processo de desenvolvimento de cinco passos:

Desenvolvimento de modelo abrangente (Análise orientada por objetos):

É um processo de desenvolvimento de sistemas que utiliza o conceito de objetos que interagem entre si e, através dessa interação, realizam tarefas computacionais. O ponto de partida para a OOA (Análise Orientada a Objetos) é criar um modelo descritivo contendo informações do projeto. Podemos chamar este artefato de proposta técnica, já que é ela que vai conter detalhes do problema a ser solucionado, objetivo do projeto, casos de uso, requisitos funcionais e não funcionais, as atividades do sistema e soluções propostas. Utilizando-se de padrões da UML (Unified Modeling Language), os requisitos do sistema são especificados, construindo artefatos necessários para a documentação. Durante todo o ciclo de vida do desenvolvimento (análise e programação), são gerados alguns diagramas que representam os objetos de análise, sendo eles: diagrama de caso de uso, de classe, de objeto, de componente, de implantação, de estado, de colaboração e de sequência.

Construção de lista de funcionalidades (Decomposição funcional):

Construção de lista de funcionalidades ou decomposição funcional se trata de uma análise de métodos onde processos complexos são divididos buscando facilitar a construção do projeto e criar uma lista das funcionalidades que deverão estar presentes no resultado final. Os processos são ordenados por prioridade de desenvolvimento, isso permite que cada parte seja analisada de forma independente e com isso é possível ter estimativas de tempo mais precisas e também identificar partes do projeto que podem contribuir para atrasos.

Planejamento por funcionalidade (Planejamento incremental):

É a etapa em que um programa ou software é entregue em partes, que são denominados de incrementos independente do seu tamanho, porém, essas partes são entregues já em sua versão final. Reduzindo o número de falhas na solução entregue ao usuário com uma determinação de esforços em cada incremento para que ambas estejam funcionando corretamente. E com isso acaba criando um ambiente rápido e prático e com agilidade em solucionar os problemas que acontecem em cada etapa.

Detalhamento por funcionalidade (Desenho orientado a objetos):

Um programador chefe irá determinar a funcionalidade que será detalhada e construída. Ele ou ela também determinará os owners e times envolvidos, enquanto definindo as prioridades dessa funcionalidade. Parte do grupo pode trabalhar na parte técnica do design enquanto outros trabalham no desenvolvimento do framework, por exemplo. Ao fim desse estágio é feito um review de design pelo time completo antes de prosseguir.

Construção por funcionalidade (Programação e teste orientado a objetos):

Nesse passo, são implementados todos os itens necessários para dar apoio ao design. Aqui, interfaces de usuário são construídas, componentes são detalhados no design técnico e um protótipo da funcionalidade é criado. A unidade é testada, inspecionada e aprovada, então a funcionalidade completa pode ser promovida para a build principal.
